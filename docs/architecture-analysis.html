2. Deep Backend Analysis

Detailed examination of the orchestrator, workflow engine, and state management patterns.

Orchestrator Engine

The GoalSeekingLoop (~1062 lines) is the central brain of the system. It replaces the earlier DeliberationOrchestrator with an adaptive iteration loop that executes up to maxIterations times.

Three Loop Modes

ModeQuality ThresholdSelf-ImprovementAdversarial ProbeCorrective Actions
standard0.75OffOffLimited (rerun_stage only)
self_improving0.60On — MetaPromptServiceOffFull 9 action types
adversarial0.60On — MetaPromptServiceOn — generates counter-evidenceFull 9 action types

Stage Execution with Quality Wrapper

Every stage passes through runStageWithQuality() which wraps execution with:

  1. Records goal_loop.stage_started event
  2. Executes the stage function
  3. Validates output against Zod schema via OutputAnalyzer
  4. Records quality report in allQualityReports
  5. Updates PromptRegistry with call metrics
  6. On failure: if self_improving mode + MetaPromptService available → attempts prompt improvement and retries once
  7. Records recovery/unrecoverable events

Exit Criteria

  • Goal achieved + decision created → STOP
  • Max iterations reached → STOP (create max_rounds_reached decision)
  • 3+ consecutive stage failures with iteration >= 2 → STOP

Self-Improvement System

The self-improvement loop is a feedback mechanism that automatically improves prompt quality during a run:

Stage fails or low quality
→
OutputAnalyzer detects issue
→
PromptRegistry.analyzeForImprovement()
→
MetaPromptService.improve()
→
Model generates improved prompt
→
PromptRegistry.improve() creates new version
→
Next iteration uses improved prompt

OutputAnalyzer Issue Detection

Detects 11 issue types: empty_output, schema_validation_failed, parse_error, zero_uuids, placeholder_content, wrong_type, missing_required_fields, invalid_enum_value, suspicious_pattern, truncated_output, excessive_length. Produces a quality score from 0 to 1.

PromptRegistry Metrics

Per-version tracking: totalCalls, successfulCalls, schemaFailures, emptyOutputs, avgQualityScore, failurePatterns (Map of pattern → count). Improvement is triggered when failure rate exceeds threshold (default 0.7) with minimum 3 calls.

P0 — CRITICAL graph.ts Enum Literal Type Mismatches
Impact: The routes/graph.ts file has pre-existing TypeScript errors where Prisma enum values (e.g., status: string) don't match expected literal types ("supported" | "contradicted" | ...). This causes type check failures across the API build.
Root cause: Prisma generates string for some enum fields; explicit type narrowing is missing.
Risk: Build failures, potential runtime type mismatches, broken CI/CD.
P0 — CRITICAL OpenAPI Schema Not Generated / Out of Sync
Impact: The spec requires schemas/openapi.yaml to cover the full API surface. No OpenAPI spec is auto-generated from the route definitions.
Risk: API contract drift between frontend and backend. No machine-readable API documentation. Spec doc says "MVP fails if schemas/openapi.yaml covers only a subset."
P0 — CRITICAL No End-to-End MVP Acceptance Test
Impact: The 14-step acceptance scenario in docs/10-mvp-acceptance.md has no automated test. The full pipeline (create project → extract claims → evidence → reviews → critiques → revision → consensus → decision) is never tested end-to-end with mock models.
Risk: Regressions can silently break the core value proposition. No confidence in system correctness.
P0 — CRITICAL Context Service — Lazy Loading Not Implemented
Impact: The spec defines a lazy loading protocol where models can request additional context mid-call. The ContextService has basic context building but no lazy loading mechanism.
Risk: Models may operate with insufficient context, producing lower-quality outputs. No mechanism to handle context window overflow gracefully.
P0 — CRITICAL Semantic Search (pgvector) Not Active
Impact: pgvector extension exists in schema (SourceEmbedding model) but semantic search is gated by EMBEDDING_ENABLED=false. No embedding generation pipeline exists. Cross-project search uses SQL LIKE instead of vector similarity.
Risk: Cross-project search quality is degraded. Spec says pgvector setup must be functional.
P1 — HIGH No Automated Lint/Format Enforcement
Impact: No ESLint or Prettier configuration in the repo. Code style drift is unchecked. The spec mentions "if lint is configured" but it isn't.
Risk: Inconsistent code style across contributors. Harder code reviews.
P1 — HIGH Web App Type Errors in Production Hooks
Impact: While we fixed many type errors in the previous session, several React Query hooks still return data: {} types due to missing generics on parseJson(res). Components use type assertions to work around this.
Risk: Runtime data access errors if response shapes change. TypeScript doesn't catch actual bugs.
P1 — HIGH Claim Dependency Graph — Frontend Route Missing
Impact: The spec defines /projects/[id]/claim-dependencies route and visual graph. The backend API exists but the frontend page is not implemented.
Risk: Feature gap visible in navigation. Users cannot visualize claim dependencies.
P1 — HIGH Argument Map (Toulmin) — No DB Model
Impact: The spec defines an ArgumentMap Prisma model for Toulmin argument exports. The model is not in the schema, and the endpoint returns hardcoded/empty data.
Risk: Feature cannot persist. Reproducibility Pack and Argument Map features are non-functional.
P1 — HIGH No CI/CD Pipeline
Impact: No GitHub Actions or CI configuration. No automated test runs, type checks, or build verification on PRs.
Risk: Regressions merge silently. No quality gate.
P1 — HIGH No E2E Browser Tests
Impact: The 7 frontend smoke tests exist but only test rendering. No Playwright/Cypress tests verify user flows (create project → start run → view decision).
Risk: UI regressions go undetected.
P1 — HIGH Rate Limiter — No Per-User Tracking
Impact: The rate limiter uses IP-based buckets. In a multi-user scenario, all requests from same IP share limits. No per-user or per-project rate limiting.
Risk: Single user can starve others. DoS vulnerability.
P1 — HIGH No Database Backup/Restore Strategy
Impact: No documented backup procedure. No pg_dump scripts. Data loss risk for long-running research projects.
Risk: Unrecoverable data loss.
P2 — MEDIUM No Docker Compose Health Checks
Impact: docker-compose.yml lacks health checks for PostgreSQL and Redis. Services may start before dependencies are ready.
Risk: Race conditions during startup. Flaky development environment.
P2 — MEDIUM No Request ID / Correlation Tracing
Impact: No unique request ID attached to API requests. Difficult to trace errors across log entries. No distributed tracing.
Risk: Harder debugging in production.
P2 — MEDIUM No Structured Error Logging
Impact: Logger exists but errors aren't consistently structured with projectId/runId context. Stack traces may leak in API responses.
Risk: Harder to diagnose production issues.
P2 — MEDIUM No Migration Rollback Strategy
Impact: Prisma migrations are forward-only. No documented rollback procedure for schema changes.
Risk: Difficult to recover from bad migrations.
P2 — MEDIUM SSE Polling Interval Fixed at 1s
Impact: The SSE endpoint polls DB every 1 second regardless of load. Under high concurrency, this creates unnecessary DB load.
Risk: Performance degradation with many concurrent runs.
P3 — LOW No API Versioning
Impact: All routes are at root level (/projects, not /api/v1/projects). No versioning strategy.
Risk: Breaking changes affect all clients simultaneously. Low risk for single-team local-first app.
P3 — LOW No Webhook / External Integration Points
Impact: No webhook system for external notifications (email, Slack, etc.).
Risk: User must actively check the UI. Low priority for MVP.
P3 — LOW No Data Export to External Formats
Impact: JSON and Markdown export exist but no CSV, BibTeX, or RIS export for academic workflows.
Risk: Minor workflow friction for academic users.
P3 — LOW No Mobile Responsive Optimization
Impact: The UI uses basic responsive breakpoints but hasn't been tested on mobile devices. Inspector panel and complex tables may not work well on small screens.
Risk: Local-first app primarily used on desktop. Low priority.
